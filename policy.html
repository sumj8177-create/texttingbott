Section 01

What Is This Tool?

The Bot Dashboard is a self-hosted, locally-run web interface for your Discord bots. It runs as a Python process on your own computer and serves a webpage only accessible from localhost — meaning no one on the internet can reach it.

Unlike SaaS Discord management tools or hosted dashboards, this tool has no backend infrastructure, no cloud database, no user accounts, and no company server that your data passes through. It is simply a local Python script that bridges your browser and Discord's API.

When you run this dashboard, the only active connections are: your browser to localhost:8080, and the bot to Discord's servers. No third party is involved.
Section 02

Zero Data Collection

We collect absolutely nothing. There is no telemetry, no crash reporting, no usage analytics, and no phone-home mechanism of any kind.

  • Bot tokens (main or custom)
  • Discord server names, IDs, or member lists
  • Channel names or IDs
  • Messages you send or read
  • Your IP address or browser fingerprint
  • Session duration or feature usage
  • Error traces or crash reports

This is not a policy that can be changed server-side. There is no server side. The Python script on your disk is the entire backend. Open it, read it, verify it yourself at any time.

Section 03

Bot Token Security

Your main bot token lives in a single place: the DISCORD_TOKEN variable at the top of discord_dashboard.py. This script never transmits your token anywhere except to Discord's own API for authentication.

Your token is never written to a log file, included in HTTP responses to your browser, sent to any server other than discord.com, or accessible via any dashboard endpoint.

Tip: Use an environment variable instead of hardcoding your token. If you ever share the file, the token won't be in it.
# More secure — use an environment variable: import os DISCORD_TOKEN = os.getenv("DISCORD_TOKEN", "") # Run with: DISCORD_TOKEN=your_token_here python discord_dashboard.py
Section 04

Custom Bot Tokens

The dashboard supports adding additional bot tokens so you can send messages through multiple bots from the same interface. Here is exactly how they are handled:

  • Tokens are validated instantly by calling Discord's /users/@me endpoint.
  • Tokens are stored in Python memory only — in a dictionary that exists only while the script is running.
  • When you close the script, all custom tokens are gone. Nothing is written to disk.
  • Custom tokens are never sent back to your browser. Only the bot's username is returned for display.
  • The token input uses type="password" so it is masked and not saved by browser autofill.
Custom bots send messages via Discord's REST API directly from the Python process. Your browser never touches the token.
Section 05

Message Content

Outbound messages travel from your browser to localhost, then from Python directly to Discord's API. They are not logged, cached, or stored anywhere on your machine.

Inbound messages (the live feed) are delivered via Server-Sent Events and held only in your browser's JavaScript memory for display. When you navigate away, they are gone.

  • Messages are never written to disk by this tool
  • Message history is fetched fresh each time you select a channel
  • Messages are not indexed, analyzed, or summarized
  • Attachments or embeds are not downloaded or stored locally
Section 06

No Analytics or Tracking

There is no analytics code anywhere in this project — no Google Analytics, no Mixpanel, no pixels, no session recordings, no heatmaps, nothing. The dashboard does not track which servers you view, which channels you open, how many messages you send, how long you use it, or what errors you encounter.

The only reason analytics are omitted is simple: this tool has no author receiving data. It is a local script. There is no company, no SaaS, no product team gathering insights. You run it; you own it.

Section 08

Session-Only Memory

Nothing persists between runs of the Python script. Every restart begins with a completely clean slate — custom bot tokens, active connections, message history, notification counts, selected server and channel. All are gone when the process exits.

The Python process holds data in three in-memory structures: the bot object, the SSE connections dictionary, and the custom bots dictionary. All three vanish when the process exits.

Section 09

Open Source Philosophy

This dashboard is distributed as readable Python and HTML files. There are no compiled binaries, no obfuscated code, no minified scripts from an unauditable CDN. Every line of code that runs on your machine is readable by you.

We believe any tool handling bot tokens must be auditable. You should never have to trust a privacy policy — you should be able to read the code and verify claims yourself.

  • Open discord_dashboard.py in any text editor
  • Search for aiohttp.post — verify every call goes to discord.com only

Section 11

Security Recommendations

  • Use environment variables instead of hardcoding your token.
  • Add the Python file to .gitignore to prevent accidental commits.
  • Regenerate your token immediately if you ever suspect it was exposed.
  • Give bots minimum permissions — Send Messages is all that is needed.
  • Do not share your screen while the dashboard is open.
  • Consider a firewall rule blocking port 8080 from all interfaces except loopback.
Never share a screenshot of your terminal or Python file without removing your bot token. Token exposure is the most common way Discord bots get hijacked.
Section 12

Technical Architecture

The web server and Discord bot share a single asyncio event loop — no threads, no race conditions, no locks needed. Server-Sent Events (SSE) are used for the live message feed: one-directional, simple, and automatically reconnecting.

# Data flow: Browser (localhost:8080) | HTTP GET/POST, SSE aiohttp web server (same asyncio loop as bot) |-- discord.py bot --> Discord Gateway (WebSocket) | on_message() --> SSE queue --> browser |-- extra_bots --> Discord REST API (per message)
Section 13

Your Rights & Full Control

  • Stop it anytime — press Ctrl+C and everything shuts down immediately.
  • Delete it completely — delete the files and every trace is gone. No account to close, no data to request deletion of.
  • Modify it freely — it is your copy. Change the port, add features, remove things you do not need.

There is no GDPR request process, no right-to-erasure procedure, no data export — because there is no data. The moment you stop the Python process, nothing about your usage exists anywhere except in Discord's own servers.

Section 14

Frequently Asked Questions

Can someone on my local network access the dashboard?
The dashboard binds to 127.0.0.1 (loopback only). Only processes on your own machine can connect. Devices on your local network cannot access it.
Does the bot log messages in the servers it is in?
No. The bot only processes messages to forward them to the SSE stream for your browser. Messages are held in the queue for a fraction of a second. If no browser tab is connected, the messages are dropped entirely.
What happens to custom bot tokens when I restart?
They are permanently gone. Custom tokens are stored in a Python dictionary in RAM. When the process exits, that memory is released. There is no file, no database, no registry entry. You will need to re-add custom bots each time you start the dashboard.
What does the dashboard do with the token I type into the Add Bot form?
Your browser sends the token over localhost to the Python backend. The backend immediately calls Discord to validate it. If valid, it stores the token in memory and returns only the bot username to your browser. The token never travels anywhere except localhost and Discord.
How can I verify this policy is actually true?
Read the Python file. Search for every outbound network call and verify each one goes to discord.com only. Look for any file write operations — there are none. Check logging usage. The code is the ultimate source of truth.
